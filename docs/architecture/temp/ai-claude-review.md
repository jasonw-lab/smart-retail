## 📋 エグゼクティブサマリー

### レビュー総評
本要件定義は、**無人スーパー運営を前提とした実務志向の管理画面**として、非常に明確な設計思想と実現可能な要件を提示している。転職活動における技術力アピールの観点から、以下の点で高い評価を得られる内容である：

#### ✅ 優れている点
1. **明確なビジネス価値** - EC依存ではなく、実店舗運営の課題解決に焦点
2. **実務志向の設計** - KPI定義、アラート運用フロー、ロール分離が具体的
3. **技術的実現可能性** - 既存の技術スタック（Vue.js + Spring Boot）で段階実装可能
4. **拡張性への配慮** - センサー連携、AI分析など将来の拡張を見据えた設計

#### ⚠️ 改善・強化が推奨される点
1. **店舗稼働監視機能の実装不足** - 機器異常検知、通信状態監視の実装が未着手
2. **アラート運用の未成熟** - 状態遷移（ON_SITE含む）の実装が不完全
3. **売上データとPOS連携** - 売上テーブルは実装済みだが、POS/決済端末との連携仕様が不明確
4. **棚卸差異（ロス）監視** - センサーデータとの統合が未実装

**アピールポイント**: ステートマシン設計、運用フロー設計力

```yaml
現状の問題:
  - Alert.resolved (Boolean) のみで、状態遷移が不十分
  - ON_SITE（現地対応中）状態が実装されていない

必要な実装:
  - データベース拡張:

      - PUT /api/v1/retail/alerts/{id}/acknowledge
      - PUT /api/v1/retail/alerts/{id}/assign
      - PUT /api/v1/retail/alerts/{id}/status
      - GET /api/v1/retail/alerts/timeline  (ダッシュボード用)

技術的ハイライト:
  - 状態遷移をEnumと権限制御で厳密化
  - SLA管理（NEW→ACK は30分以内など）
  - メール/Slack通知統合
```

**転職面接での訴求ポイント**:
> 「アラート運用の成熟度を高めるため、NEW→ON_SITE→CLOSEDまでの6段階の状態遷移を実装し、各段階での権限制御とSLA管理を組み込みました。これにより、巡回担当者が現地対応中か本部で調査中かを可視化し、対応漏れを防止する体制を構築しました。」

### 2. 技術面接での訴求ポイント

#### Q: 「このプロジェクトで最も苦労した技術的課題は？」
**回答例**:
> 「無人店舗では通信断が頻発するため、Edge側でのデータ一時保存と復旧後の再送（Store & Forward）が必須でした。バックエンド側では遅延到着データを正しい時系列で処理し、重複登録を防ぐ冪等性保証の実装に工夫が必要でした。結果として、Redis の分散ロック と MySQL のユニーク制約を組み合わせた仕組みで解決しました。」

#### Q: 「このシステムのスケーラビリティはどう確保しますか？」
**回答例**:
> 「現在は REST API ベースですが、店舗数が1000を超える規模を想定し、イベント駆動アーキテクチャへの移行を設計しています。具体的には、RabbitMQ/Kafka でアラート発生イベントをパブリッシュし、通知サービス・分析サービスが非同期にサブスクライブする構成です。また、読み取りクエリはRead Replicaへルーティングし、書き込みとの負荷分散を実現します。」

#### Q: 「なぜ EC ではなく実店舗に特化したのですか？」
**回答例**:
> 「無人小売業界では、Amazon Go や ファミリーマート の無人店舗が急成長しており、現場運営の効率化が競争力の源泉だからです。EC管理画面は既に多くのOSSや商用製品がありますが、IoT機器監視・ロス管理・現場オペレーション支援を統合した管理画面は差別化ポイントが高いと判断しました。」

```yaml
  - 操作ログの記録（sys_log テーブル）

  - DB接続情報の環境変数化
  - API キーの暗号化保存
  - HTTPS 通信必須

  - 重要操作（価格変更、在庫調整）の監査ログ保存期間: 3年
```

#### 1.3 可用性・障害対応
```yaml

  - MTTR（平均復旧時間）: 30分以内

バックアップ:
  - DB 日次フルバックアップ（3世代保持）
  - トランザクションログのリアルタイムバックアップ

障害時対応:
  - ダッシュボードが停止しても POS 売上は継続可能

```

```yaml
    - 賞味期限接近（3日前）
    → 対応SLA: 4時間以内に確認、当日中に対応
  
  LOW (P4):
    - 在庫過多
    - 定期棚卸通知
    → 対応SLA: 24時間以内に確認
```
