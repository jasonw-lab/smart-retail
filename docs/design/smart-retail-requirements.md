# Smart Retail（無人スーパー向け）管理画面 要件定義 v1.1

> **本バージョンについて**: デモ・PoC実装を目的とした要件定義です。
> **Phase 1（コア機能）** に集中し、最小限の機能で無人店舗運営の価値を実証します。
> セキュリティ・顧客対応・緊急時対応などの本番運用に必須の要件は含まれていません。

---

## 1. 背景・目的

本管理画面は、**無人スーパー・省人化小売店舗の現場運営を支援するための管理画面**である。

EC（ネット注文）を前提とせず、以下を主目的とする。

* 無人店舗の **稼働状況・売上・在庫状態を横断的に把握** する
* 現場対応が必要な **異常（在庫・期限・機器）を早期に検知** する
* 補充・値引き・保守対応など **実店舗オペレーションの判断を支援** する

本要件定義は、実務を想定した Smart Retail システム設計力を示すことを目的とする。

## 2. コア機能（Phase 1）の設計思想

### 2.1 設計の基本理念

本管理画面は、
**「売上を見るための画面」ではなく、
「無人店舗を安全かつ効率的に運営するための判断基盤」**として設計されている。

#### 2.1.1 なぜ Phase 1 に集中するか

無人店舗運営において最も重要なのは：
1. **店舗が正常に稼働しているか**を把握すること
2. **異常が発生したら即座に検知**すること
3. **現場対応の優先順位を明確化**すること

高度な分析・予測・最適化は、まず基本的な監視・検知が安定稼働してから意味を持つ。

#### 2.1.2 Phase 1 の3つの柱

| 柱 | 目的 | 対応するユーザー行動 |
|----|------|-------------------|
| **監視** | 店舗状態を横断的に把握 | ダッシュボードで全店状況を確認 |
| **検知** | 異常を早期発見 | アラートで問題を即座に認識 |
| **対応** | 現場オペレーションを支援 | 在庫・商品管理で補充・値引き判断 |

## 4. 想定ロールと責務

### 4.1 ロール定義（Phase 1 対象）

* **本部運営管理者**
  全店舗横断の KPI 監視、異常検知、運営基準の統制を行う

* **店舗オペレーター**
  担当店舗の在庫補充、棚割・入出庫、現地対応を行う
  * 運用前提: 1人あたり5-10店舗を担当、1日1-2回巡回
  * 対応優先度に応じて即時駆けつけ or 次回巡回時対応

### 4.2 ロール別責務一覧（Phase 1）

| ロール | 主な閲覧 | 主な操作 |
|--------|---------|---------|
| 本部運営管理者 | 全店 KPI / 全アラート | 基準値設定、全店横断確認 |
| 店舗オペレーター | 自店舗在庫 / 状態 | 補充記録、アラート対応 |

## 5. アラート機能詳細設計

### 5.1 アラート優先度定義

| 優先度 | 分類 | 例 | 対応目標時間 | 表示色 |
|--------|------|-----|-------------|--------|
| P1 | 緊急 | 在庫切れ（主力商品）| 4時間以内 | 🔴 赤 |
| P2 | 高 | 賞味期限接近（3日以内）| 24時間以内 | 🟠 橙 |
| P3 | 中 | 在庫過多、低回転商品 | 週次確認 | 🟡 黄 |
| P4 | 低 | 情報通知（参考情報）| 任意 | ⚪ 灰 |

### 5.2 アラート種別と検出ロジック

#### 5.2.1 在庫切れアラート（LOW_STOCK）

| 項目 | 内容 |
|------|------|
| **検出契機** | 定期バッチ（1時間毎）または在庫更新時 |
| **検出条件** | 店舗×商品の **有効在庫合計**（Σ 有効ロットの `quantity`）≤ `product.reorder_point` |
| **優先度判定** | 主力商品（売上上位20%）: P1 / その他: P2 |
| **アラート内容** | 店舗名、商品名、現在在庫数、発注点 |

> **有効ロット**: `quantity > 0` かつ `expiry_date >= CURRENT_DATE`（期限切れは除外）

**例：在庫切れアラート**
```
[P1] 在庫切れ警告
店舗: 東京本店
商品: プレミアムコーヒー（SKU: PRD-001）
現在在庫: 2個
発注点: 10個
検知日時: 2026-01-28 10:30:00
```

#### 5.2.2 賞味期限接近アラート（EXPIRY_SOON）

| 項目 | 内容 |
|------|------|
| **検出契機** | 日次バッチ（毎日 06:00）|
| **検出条件** | `quantity > 0` かつ `expiry_date >= CURRENT_DATE` のロットで、`expiry_date - CURRENT_DATE <= 7日` |
| **優先度判定** | P1: 残日数 ≤ 1日 / P2: 残日数 2-3日 / P3: 残日数 4-7日 |
| **アラート内容** | 店舗名、商品名、ロット番号、賞味期限日、残日数 |

**例：賞味期限接近アラート**
```
[P2] 賞味期限接近
店舗: 横浜駅前店
商品: オーガニックティー（SKU: PRD-002）
ロット: LOT-2026-0125
賞味期限: 2026-01-31
残日数: 3日
検知日時: 2026-01-28 06:00:00
推奨対応: 値引き販売または店舗間移動を検討
```

#### 5.2.3 在庫過多アラート（HIGH_STOCK）

| 項目 | 内容 |
|------|------|
| **検出契機** | 週次バッチ（毎週月曜 09:00）|
| **検出条件** | 店舗×商品の **有効在庫合計**（Σ 有効ロットの `quantity`）≥ `product.max_stock_level * 1.5` |
| **優先度** | `product.shelf_life > 0`（賞味期限あり）: P2 / それ以外: P3 |
| **アラート内容** | 店舗名、商品名、現在在庫数、適正在庫上限 |

**例：在庫過多アラート**
```
[P3] 在庫過多
店舗: 名古屋栄店
商品: ミネラルウォーター（SKU: PRD-003）
現在在庫: 150個
適正上限: 100個
超過率: 50%
検知日時: 2026-01-27 09:00:00
推奨対応: 発注抑制または他店舗への移動を検討
```

### 5.3 アラート状態遷移

```
NEW → ACK → IN_PROGRESS → RESOLVED → CLOSED
```

| 状態 | 説明 | 遷移操作 | 操作者 |
|------|------|---------|--------|
| **NEW** | システム自動検知 | 自動生成 | システム |
| **ACK** | 管理者が確認済み | 「確認」ボタン | 本部管理者 |
| **IN_PROGRESS** | 対応中 | 「対応開始」ボタン | 店舗オペレーター |
| **RESOLVED** | 対応完了 | 「完了」ボタン + 対応内容入力 | 店舗オペレーター |
| **CLOSED** | 検証完了・クローズ | 「クローズ」ボタン | 本部管理者 |

### 5.4 アラート検出の実装方針

```

│                    アラート検出フロー                          │

│                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │ 在庫更新     │───▶│ 在庫チェック  │───▶│ アラート生成  │  │
```

### 5.5 重複アラートの抑制

同一条件のアラートが繰り返し生成されることを防ぐため：

| ルール | 説明 |
|--------|------|
| **同一商品・同一種別** | 未解決アラートが存在する場合は新規生成しない |
| **状態昇格** | 条件悪化時は既存アラートの優先度を上げる |
| **自動クローズ** | 条件解消検知後、`IN_PROGRESS` のみ自動で `RESOLVED` へ遷移し、条件解消後24時間で自動 `CLOSED`（`NEW/ACK` は対象外） |

## 6. KPI 定義（Phase 1）

| KPI | 定義 | 算出式 |
|-----|------|--------|
| 本日売上 | 店舗 POS / 決済端末集計売上 | Σ(各店舗の売上) |
| 稼働中店舗数 | 正常稼働している店舗数 | 総店舗数 - 異常店舗数 |
| 在庫切れ SKU 数 | 在庫が発注点以下の商品数 | Σ(在庫数 <= 発注点の SKU) |
| 賞味期限接近数 | 期限3日以内の商品数 | Σ(期限残 <= 3日の SKU) |
| アクティブアラート数 | 未対応アラート数 | Σ(状態 = NEW or ACK or IN_PROGRESS) |

**異常店舗の判定基準（Phase 1）**:
- `store.status` が `MAINTENANCE` または `STOPPED`（停止中）の店舗を「異常店舗」とする
- 例：店舗一覧（S-01）の表示が「🟡メンテ」「🔴停止中」に該当

> **Store has many Devices**: 店舗は複数のデバイス（決済端末、カメラ、ゲート、冷凍庫センサー等）を持つ。
> アラートは「どの機器(Device ID)」で発生したかを特定可能とし、オペレーターの初動対応時間を短縮する。

#### 8.1.1 ロット（賞味期限）管理方針

同一商品（同一SKU）でも賞味期限が異なるケースを扱うため、在庫は **ロット単位**で保持する。

- **基本単位**: 1在庫レコード = 店舗 × 商品 × ロット（=入荷/補充バッチ）
- **賞味期限の扱い**: 同一SKUでも賞味期限が異なれば **別ロットとして別レコード**で管理する（`lot_number`, `expiry_date` を保持）
- **画面表示（I-01）**: 一覧はSKU単位で集約（合計在庫数 + 最古の賞味期限）し、必要時にロット別明細へ展開する
- **在庫引当（減算）ルール**:
  - ロットが指定できる出庫（廃棄/移動など）は **指定ロット**から減算
  - 売上連携などでロット指定が無い出庫は **FEFO（First-Expire-First-Out）** を基本とし、賞味期限が近いロットから優先的に減算する（必要に応じて複数ロットにまたがって減算する）
  - **期限切れロット（`expiry_date < CURRENT_DATE`）はFEFO対象外**（Phase 1）
  - FEFOの同率（同一賞味期限）の場合は入庫日時などでFIFOにフォールバックする
- **入庫（補充）ルール（Phase 1）**:
  - `product.shelf_life > 0`（賞味期限あり）の商品は、補充時に `expiry_date` 入力を必須とする
  - 同一店舗×同一商品×同一 `expiry_date` のロットが存在する場合は、当該ロットに加算する（存在しない場合は新規ロット作成）
- **期限切れ在庫の扱い**（Phase 1）:
  - 画面（I-01）で期限切れロットを含むSKUは「🔴期限切れ」状態で表示（最優先）
  - 廃棄または調整による在庫ゼロ化を推奨（I-01 で対象ロットを指定して「廃棄（在庫調整）」登録できる）
  - Phase 2でEXPIREDアラート（P1）の自動生成を検討
- **アラート**:
  - 在庫切れ（LOW_STOCK）はSKU集約量（合計在庫）で判定
  - 賞味期限接近（EXPIRY_SOON）はロット単位で判定し、通知にはロット番号と賞味期限を含める

### 8.2 アラートテーブル定義

```sql
CREATE TABLE alert (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    store_id BIGINT NOT NULL,
    product_id BIGINT,
    device_id BIGINT,
    lot_number VARCHAR(50),
    alert_type ENUM('LOW_STOCK', 'EXPIRY_SOON', 'HIGH_STOCK', 'COMMUNICATION_DOWN', 'PAYMENT_TERMINAL_DOWN') NOT NULL,
    priority ENUM('P1', 'P2', 'P3', 'P4') NOT NULL,
    status ENUM('NEW', 'ACK', 'IN_PROGRESS', 'RESOLVED', 'CLOSED') DEFAULT 'NEW',
    message VARCHAR(500),
    threshold_value VARCHAR(50),
    current_value VARCHAR(50),
    detected_at DATETIME NOT NULL,
    acknowledged_at DATETIME,
    resolved_at DATETIME,
    closed_at DATETIME,
    resolution_note TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (store_id) REFERENCES store(id),
    FOREIGN KEY (product_id) REFERENCES product(id),
    FOREIGN KEY (device_id) REFERENCES device(id)
);
```

### 8.3 デバイステーブル定義

> **設計方針**: Phase 1 から Device エンティティを導入し、機器別のアラート発生源を特定可能にする。
> これにより「POSの紙切れ」と「カメラの不調」を即座に切り分け、オペレーターの初動対応時間を短縮する。
>
> **Phase 1での扱い**: Device は保持するが、Heartbeat監視による自動アラート生成（`COMMUNICATION_DOWN` 等）は Phase 2 で実装する。

```sql
CREATE TABLE device (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    store_id BIGINT NOT NULL,
    device_code VARCHAR(50) NOT NULL UNIQUE,
    device_type ENUM('PAYMENT_TERMINAL', 'CAMERA', 'GATE', 'REFRIGERATOR_SENSOR', 'PRINTER', 'NETWORK_ROUTER') NOT NULL,
    device_name VARCHAR(100) NOT NULL,
    status ENUM('ONLINE', 'OFFLINE', 'ERROR', 'MAINTENANCE') DEFAULT 'ONLINE',
    last_heartbeat DATETIME,
    error_code VARCHAR(50),
    metadata JSON,  -- デバイス固有の追加情報（モデル名、シリアル番号等）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (store_id) REFERENCES store(id),
    INDEX idx_device_store (store_id),
    INDEX idx_device_status (status),
    INDEX idx_device_heartbeat (last_heartbeat)
);
```

#### 8.3.1 デバイスタイプ定義

| device_type | 説明 | 監視項目 |
|-------------|------|----------|
| PAYMENT_TERMINAL | 決済端末（POS） | 稼働状態、カードリーダー接続、最終決済日時 |
| CAMERA | 監視カメラ / AIカメラ | 映像取得状態、AI処理状態 |
| GATE | 入退店ゲート | 開閉状態、センサー反応 |
| REFRIGERATOR_SENSOR | 冷凍庫/冷蔵庫センサー | 温度、ドア開閉 |
| PRINTER | レシートプリンター | 用紙残量、紙詰まり |
| NETWORK_ROUTER | 店舗ネットワーク機器 | 接続状態、遅延 |

### 8.4 在庫トランザクションテーブル定義（Phase 1）

在庫はロット単位で管理し、入出庫の履歴は **ロット単位で差分（増減）を記録**する。
（ロット未指定の売上は FEFO により複数ロットへ按分される可能性がある）

```sql
CREATE TABLE inventory_transaction (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    inventory_id BIGINT NOT NULL,
    transaction_type ENUM('SALE', 'STOCK_IN', 'ADJUSTMENT', 'DISCARD', 'TRANSFER') NOT NULL,
    quantity_change INT NOT NULL,  -- マイナスが減算
    reference_id VARCHAR(100),  -- 売上ID、補充ID等
    note VARCHAR(255),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (inventory_id) REFERENCES inventory(id),
    INDEX idx_inventory_tx_inventory (inventory_id),
    INDEX idx_inventory_tx_type (transaction_type),
    INDEX idx_inventory_tx_created_at (created_at)
);
```

## 11. まとめ

### 11.1 Phase 1 の価値

Phase 1（コア機能）は、無人店舗運営において**最も基本的かつ不可欠な機能**を提供する：

1. **即座に全店状況を把握**できる（ダッシュボード）
2. **問題を見逃さない**（アラート自動検知）
3. **現場対応の優先度が明確**（優先度別アラート管理）
4. **在庫・商品を適切に管理**できる（基本的な CRUD 操作）

### 11.2 実装指針

* まず Phase 1 を完全に動作させる
* Phase 1 が安定したら Phase 2 の機能を段階的に追加
* 各機能は独立してテスト可能な設計とする
