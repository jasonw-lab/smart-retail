## 改善推奨事項

### 1. ロット管理の実装詳細に不足あり【高優先度】

**問題点：**
> 「売上連携などでロット指定が無い出庫は FEFO を基本とし...」

実店舗のPOS売上では、通常「どのロットから売れたか」を特定できません。

**推奨対応：**
```yaml
売上減算のロット振り分けロジック:
  1. 売上トランザクション受信時、商品SKUの在庫ロット一覧を取得
  2. 有効在庫（expiry_date >= CURRENT_DATE）を賞味期限の昇順でソート
  3. 先頭ロットから順に在庫を減算
  4. 在庫不足の場合は次のロットから減算（複数ロットにまたがる可能性あり）
  5. 減算履歴をinventory_transactionテーブルに記録
```

**追加テーブル案：**
```sql
CREATE TABLE inventory_transaction (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    inventory_id BIGINT NOT NULL,
    transaction_type ENUM('SALE', 'STOCK_IN', 'ADJUSTMENT', 'DISCARD', 'TRANSFER') NOT NULL,
    quantity_change INT NOT NULL,  -- マイナスが減算
    reference_id VARCHAR(100),  -- 売上ID、補充ID等
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (inventory_id) REFERENCES inventory(id)
);
```

### 2. 在庫切れアラートの発注点判定が曖昧【高優先度】

**問題点：**
> 「検出条件: `inventory.quantity <= product.reorder_point`」

**課題：**
- ロット単位で管理しているのに、どのロットの在庫数で判定するのか不明
- 複数ロットの合計在庫数で判定する場合、その旨を明記すべき

**推奨修正：**
```
検出条件: 
  店舗×商品の有効在庫合計（Σ有効ロットの quantity）<= product.reorder_point
  ※有効在庫: expiry_date >= CURRENT_DATE のロット
```

### 3. 賞味期限接近アラートの優先度判定に矛盾【高優先度】

**問題点：**
セクション5.2.2では：
> 「閾値設定: P1: 1日以内 / P2: 3日以内 / P3: 7日以内」

セクション5.1では：
> 「P2 | 高 | 賞味期限接近（3日以内）」

**推奨対応：**
閾値設定を明確化し、一貫性を保つ：
```
EXPIRY_SOON アラート優先度:
- P1 (緊急): 残日数 <= 1日
- P2 (高): 残日数 2-3日
- P3 (中): 残日数 4-7日
```

### 4. Device エンティティの監視ロジックが未定義【中優先度】

**問題点：**
Device テーブルに `last_heartbeat` フィールドがあるが、どのように更新され、どのタイミングでアラートを発生させるかが未定義。

**推奨対応：**
```yaml
Deviceアラート検出ロジック（Phase 2 候補）:
  検出契機: 定期バッチ（5分毎）
  検出条件: 
    - last_heartbeat が 15分以上前 → COMMUNICATION_DOWN (P1)
    - device.status = 'ERROR' → デバイス種別に応じたアラート (P1/P2)
  アラート内容: 店舗名、デバイス名、デバイスタイプ、最終接続日時
```

**Phase 1での扱いを明記：**
```
※Phase 1ではデバイステーブルは準備するが、アラート自動検出は Phase 2 で実装。
　Phase 1ではデバイス情報の手動登録・参照のみサポート。
```

### 5. 在庫過多アラート（HIGH_STOCK）の実用性に疑問【中優先度】

**問題点：**
無人店舗運営において、在庫過多は即座に対応が必要な問題ではなく、P3（週次確認）は適切。ただし：

- 賞味期限付き商品の在庫過多は期限切れリスク（廃棄コスト）に直結
- 常温日用品の在庫過多は資金効率の問題

**推奨対応：**
```
HIGH_STOCK アラートの優先度判定を商品特性で分岐:
- 賞味期限付き商品（shelf_life > 0）の過剰在庫 → P2（早期値引き/移動検討）
- 賞味期限無し商品の過剰在庫 → P3（次回発注調整）
```

---

### 6. KPI定義の「稼働中店舗数」が曖昧【中優先度】

**問題点：**
> 「稼働中店舗数: 正常稼働している店舗数 = 総店舗数 - 異常店舗数」

**何をもって「異常」とするか**の定義が必要：
```
異常店舗の判定基準（Phase 1）:
- store.status = 'CLOSED' または 'MAINTENANCE'
- （Phase 2）決済端末が15分以上オフライン
```

---

### 7. アラート自動クローズのロジックが不明確【中優先度】

**問題点：**
> 「条件解消後24時間経過で自動的にRESOLVEDへ遷移」

**課題：**
- 「条件解消」をどう判定するか（在庫補充されたら即時？）
- RESOLVEDへの遷移は人手で行うべき（オペレーターが対応完了を確認）

**推奨修正：**
```
自動クローズルール:
1. 条件解消検知（在庫補充で reorder_point 以上になった等）
2. アラートの status が IN_PROGRESS → 自動的に RESOLVED へ遷移
3. 条件解消後24時間経過 → 自動的に CLOSED（管理者承認不要）
※ NEW/ACK 状態のアラートは自動クローズ対象外（放置検知のため）
```

---

### 8. ロット別在庫の入庫ルール（Phase 1で必要）【高優先度】

**問題点：**
現在、減算ロジックは定義されているが、**入庫（補充）時のロット生成ルール**が未定義。

**推奨対応：**
```yaml
在庫補充時のロット生成:
  入力項目: 商品SKU, 補充数量, 賞味期限日（任意）
  処理:
    1. 賞味期限が指定された場合:
       - 同一店舗×商品×賞味期限のロットが存在 → 既存ロットに加算
       - 存在しない → 新規ロット作成（lot_number 自動採番）
    2. 賞味期限が未指定の場合:
       - shelf_life > 0 の商品 → 入力必須エラー
       - shelf_life = 0 の商品 → 賞味期限なしロットとして登録
```
